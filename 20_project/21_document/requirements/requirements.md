# requirements

## 1. 目的
本ステージ（requirements）では、レジャー向け節約ドライブアプリ「ドライブ・コンパス」に対する要求を、基本設計で機能分解・責務分担できる粒度の要件へ整理する。特に、時間効率（タイパ）と移動コスト節約を同時比較できる体験を中核価値とし、正常系と異常系の双方を明示した要件として定義する。

## 2. 入力
- 入力資料: 00_request/minutes_v100.md
- 開発モード: new（新規開発）
- 前提ドメイン要件:
  - 3ルート比較（最速／賢く節約／完全節約）
  - 比較軸（料金／時間／燃料）
  - ガソリン価格3地点（出発地／途中／到着地）
  - 差額の体験価値化
  - 段階導入（MVP→拡張）

## 3. 全体構成
- 対象ステージ: requirements
- 成果物: 20_project/21_document/requirements/requirements.md
- 要件分類:
  - 機能要件（ルート比較、費用可視化、体験価値提案）
  - データ要件（ルート、料金、燃料価格、算出結果）
  - 異常系要件（API失敗、データ未取得、入力不備）
  - 段階導入要件（MVPから段階拡張）
- トレーサビリティ方針:
  - 全要件にReq-IDを付与する
  - 正常系/異常系シナリオと受け入れ条件をReq-IDで対応づける

## 4. 詳細
### 4.1 要件一覧（Req-ID付き）
- REQ-01 ルート比較表示
  - 出発地と到着地を入力した際、3つの候補ルート（最速、賢く節約、完全節約）を同一画面で比較可能であること。
  - 各候補に対して所要時間、高速料金、推定ガソリン代、合計移動コストを表示すること。

- REQ-02 ルート定義の一貫性
  - 最速は高速道路優先、賢く節約は部分高速利用、完全節約は高速不使用として算出すること。
  - 各ルートの算出条件が利用者に識別できること。

- REQ-03 料金・時間・燃料の比較可能性
  - 3ルートの比較軸として、時間、料金（高速料金）、燃料コスト（推定ガソリン代）の3要素を必須表示すること。
  - 合計移動コストは料金と燃料コストの合算値として提示すること。

- REQ-04 ガソリン価格3地点可視化
  - 出発地、途中地点、到着地の3地点について、価格情報を取得できる場合はそれぞれ表示すること。
  - 3地点の価格差を基に、給油判断に資するメッセージを提示可能であること。

- REQ-05 差額の体験価値化
  - ルート間で生じる移動コスト差額を、レジャー体験（例: 食事、スイーツ等）に換算した文言で提示できること。
  - 差額ゼロまたは負値となる場合の表示ルールを定義し、誤解を招かないこと。

- REQ-06 段階導入（フェーズ適用）
  - フェーズ1（MVP）: 出発地/到着地入力に対して、所要時間および高速料金を表示すること。
  - フェーズ2: 推定ガソリン代を追加すること。
  - フェーズ3: 3地点のガソリン価格比較を追加すること。
  - フェーズ4: 差額の体験価値化を追加すること。

- REQ-07 入力バリデーション
  - 出発地/到着地が未入力または無効な場合、算出を開始せず、修正可能なエラーメッセージを表示すること。

- REQ-08 API失敗時の継続性
  - ルート取得APIまたは料金/価格取得APIが失敗した場合、処理全体を即時中断せず、取得済み情報で表示継続できること。
  - 失敗したデータ種別を利用者が判別できるメッセージを表示すること。

- REQ-09 データ未取得時の代替表示
  - 料金、燃料価格、または体験換算に必要なデータが未取得の場合、当該項目を「未取得」または同等の明示表現で表示すること。
  - 未取得項目を含む場合でも、算出可能な項目は表示し、利用者の意思決定を阻害しないこと。

- REQ-10 応答時間と再試行
  - 外部データ取得で一定時間内に応答がない場合、タイムアウトとして扱い、再試行導線を提供すること。

- REQ-11 トレーサビリティ
  - 要件、シナリオ、受け入れ条件、次工程引き継ぎ項目をReq-IDで追跡可能にすること。

### 4.2 正常系シナリオ
- SCN-N-01（対象: REQ-01, REQ-02, REQ-03）
  - 利用者が出発地と到着地を入力すると、3ルートが算出され、時間・高速料金・推定ガソリン代・合計移動コストが比較表示される。

- SCN-N-02（対象: REQ-04）
  - 3地点のガソリン価格が取得でき、地点別価格と給油判断メッセージが表示される。

- SCN-N-03（対象: REQ-05）
  - 最速ルートと節約系ルートの差額が算出され、体験価値換算メッセージが表示される。

- SCN-N-04（対象: REQ-06）
  - フェーズ進行に従い、MVP機能から順次拡張機能が追加され、既存機能の表示整合性が維持される。

### 4.3 異常系シナリオ
- SCN-E-01（対象: REQ-07）
  - 出発地または到着地が未入力の場合、算出が実行されず、入力修正を促す。

- SCN-E-02（対象: REQ-08, REQ-10）
  - ルートAPI呼び出しが失敗またはタイムアウトした場合、利用者へ失敗理由を通知し、再試行可能な状態を提示する。

- SCN-E-03（対象: REQ-08, REQ-09）
  - ガソリン価格APIが一部地点で失敗した場合、取得済み地点のみ表示し、未取得地点は未取得表示とする。

- SCN-E-04（対象: REQ-09）
  - 高速料金データが未取得の場合、料金項目は未取得表示とし、時間や他の算出可能項目は表示を継続する。

- SCN-E-05（対象: REQ-05, REQ-09）
  - 体験価値換算に必要な単価データが取得できない場合、差額金額のみ表示し、体験換算表示は抑止する。

### 4.4 受け入れ条件
- AC-01（REQ-01, REQ-03）
  - 3ルートが同一画面で比較でき、各ルートに時間・高速料金・推定ガソリン代・合計移動コストが表示されること。

- AC-02（REQ-02）
  - 3ルートの算出条件（高速優先/部分高速/高速不使用）が利用者に判別可能であること。

- AC-03（REQ-04）
  - 出発地・途中・到着地の価格が取得できた地点について個別表示されること。

- AC-04（REQ-05）
  - ルート差額が体験価値メッセージとして表示され、差額ゼロまたは負値時は誤解のない表示ルールが適用されること。

- AC-05（REQ-06）
  - フェーズ1から4までの機能追加順序が要件として定義され、後工程が段階的に実装計画へ落とし込めること。

- AC-06（REQ-07）
  - 必須入力欠落時に算出処理が開始されず、修正可能なエラーメッセージが表示されること。

- AC-07（REQ-08, REQ-09）
  - API失敗またはデータ未取得時、失敗項目が明示され、取得済み情報での表示継続が可能であること。

- AC-08（REQ-10）
  - タイムアウト時に再試行導線が提供されること。

- AC-09（REQ-11）
  - 要件、シナリオ、受け入れ条件、引き継ぎ項目がReq-IDで相互参照できること。

## 6. 課題・リスク
- 外部API依存リスク
  - ルート、料金、ガソリン価格の各データ取得先に障害や仕様変更が発生した場合、表示欠損や比較精度低下が生じる。

- データ鮮度・整合性リスク
  - 3地点価格の更新タイミング差により、実運用価格との乖離が発生する可能性がある。

- 概算値の説明責任
  - 高速料金や燃料費が概算の場合、利用者期待とのズレが不満要因になるため、概算である旨の明示が必要。

- 体験価値換算の妥当性
  - 換算ロジックが利用者属性に合わない場合、提案価値が低下する可能性がある。

- 段階導入時の後方互換
  - フェーズ拡張時に既存表示項目との整合が崩れると、比較体験の一貫性が損なわれる。
