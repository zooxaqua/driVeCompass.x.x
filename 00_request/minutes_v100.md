# 議事録

## 1. 基本情報
- 議題: レジャー向け節約ドライブアプリ企画整理
- アプリ名（案）: ドライブ・コンパス 〜節約とタイパの天秤〜
- 想定利用シーン: 休日のレジャー移動（家族・友人との外出）
- 技術方針: フロントエンド Next.js（React）、バックエンド Supabase

## 2. 企画コンセプト
- レジャー時の「時間効率（タイパ）」と「移動コスト節約」を同時に比較・判断できる体験を提供する。
- 単なる経路検索ではなく、移動で浮いた金額を現地体験に変換して、外出の満足度を高める。

## 3. 主要機能（こだわりポイント）

### 3-1. 3パターンのルート比較
出発地と到着地の入力に対し、以下3案を横並び表示する。

- 最速: 全行程で高速道路利用（遊ぶ時間を最大化）
- 賢く節約: 一部区間のみ高速利用（渋滞区間・峠道などを回避しつつ費用最適化）
- 完全節約: 全行程を下道利用（差額を観光・食事に回す）

比較表示の観点:
- 所要時間
- 高速料金
- 推定ガソリン代
- 合計移動コスト

### 3-2. ガソリン価格の見える化
出発地・途中・到着地の3地点で価格情報を提示し、給油判断を支援する。

- 出発地: 出発前に満タン給油すべきかを判断
- 途中（高速道路上）: SA給油が割高な場合に注意喚起（例: ここで入れると割高）
- 到着地: 現地給油の方が安い場合、帰路給油計画を提案

### 3-3. 浮いたお金の使い道提案
高速利用案と節約案の差額を、体験価値に変換して表示する。

表示例:
- 差額 3,000円の場合
- この差額で現地の名物スイーツ2人分、またはランチ2人分が実質無料

狙い:
- 節約の実感を強化
- レジャー気分を高める
- ルート選択の納得感を向上

## 4. 実現に向けた技術メモ

### 4-1. ルートと所要時間
- Google Maps API を利用し、高速利用ルートと高速回避ルートの取得・比較を行う。

### 4-2. 高速料金
- 日本の高速料金は計算条件が複雑なため、段階的に対応する。
- 候補:
	- NEXCO（ドラぷら）等の情報参照
	- 簡易ロジックによる概算計算
	- 既存の料金検索API（例: ナビタイム等）の検討

### 4-3. ガソリン価格
- 本格対応:
	- 政府統計データ（e-Stat API）や価格比較情報の活用
- 初期対応:
	- Google Maps API のガソリンスタンド情報（価格が取れる地点）を活用

## 5. 開発ロードマップ（段階導入）
挫折しにくいよう、機能を段階的に追加する。

1. MVP: 出発地・目的地入力で所要時間と高速代を表示
2. 次段階: ガソリン代の概算計算機能を追加
3. 拡張段階: 地点別ガソリン価格比較を追加
4. 仕上げ: 差額の使い道提案（体験価値表示）を追加

## 6. 期待価値・ターゲット
- ターゲット: 家族・友人とレジャー外出する層
- 提供価値:
	- 時間優先と費用優先の意思決定を明快化
	- 移動コストをレジャー価値に変換して可視化
	- 外出計画の満足度向上

## 7. 決定事項・次アクション

### 決定事項
- コンセプトは「レジャー × 節約 × タイパ」の三軸で進める。
- 開発は段階導入で進め、まずはMVP完成を優先する。
- 技術構成は Next.js（React） + Supabase を採用する。

### 次アクション
1. MVPの画面要件とAPI入出力を定義する。
2. ルート比較（最速・節約・完全節約）の算出ロジックを仮実装する。
3. 高速料金・ガソリン価格データの取得方式をPoCで比較する。

## 8. エージェント
- 開発工程ごとにエージェントを設定する。
- 新規要求の場合は新規開発として動作し、要求が差分の場合は差分開発を行う
- それぞれのエージェントは前後の工程を理解しており、後工程で利用できる形（mdファイル）でドキュメントを渡す。
- ドキュメントは各工程の成果物とし、全体構成と、差分（差分開発のみ）をそれぞれ記載する。
- すべてのエージェントは前後のエージェントと連携する。前工程のドキュメントに不明点があれば、前工程に質問として返し前工程はドキュメントを修正し再度次の工程から実効する

## 9. 開発工程
1. 要件定義 - 要求から要件を定義する。要件は基本設計に分解できる粒度にまとめる。正常系と異常系 両方の観点を持つ。
2. 基本設計 - 要件からソフト全体の構成を行う。フロントエンド、バックエンドの役割分担も決める。正常系と異常系 両方の観点を持つ。
3. 詳細設計 - フロントエンドとバックエンドぞれぞれ別で設計を行う。
4. 実装 - フロントエンドとバックエンドそれぞれ別でコーディングを行い、関数単位で正常系と異常系の単体評価も行う。
5. 結合テスト - 基本設計が満たされているかを確認する。正常系と異常系 両方の観点を持つ。
6. システムテスト - 要件が満たされているかを確認する。正常系と異常系 両方の観点を持つ。
